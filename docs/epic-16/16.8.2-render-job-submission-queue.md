# Sprint 16.8.2 — Render Job Submission & Queue Integration

Status: In Progress  
Owner: Ho Quan  
EPIC: 16 — Usable Product Integration Roadmap

## Goal

Submit the immutable `ReviewRenderContract` produced by Sprint 16.8.1 to the existing render queue without allowing the queue or render layer to access `TimelineRuntimeStore`.

## Architecture

```text
ReviewToRenderHandoffRuntime
  -> ReviewRenderContract
  -> RenderJobSubmissionRuntime
  -> RenderQueueSubmissionPort
  -> QueueServiceRenderSubmissionAdapter
  -> QueueType.RENDER_RUNTIME
```

## Scope

- Add revision- and checksum-aware render submission contracts.
- Add a queue submission port to isolate the runtime domain from SQLAlchemy/FastAPI.
- Add an adapter for the existing `QueueService`.
- Serialize a deep-copied handoff contract into the queue payload.
- Generate a deterministic idempotency key from production, snapshot, and checksum.
- Return the same queue job for duplicate submissions within the adapter lifecycle.
- Reject tampered or invalid contracts before queue creation.
- Preserve all 16.8.1, timeline, history, manual editing, and render architecture regressions.

## Definition of Done

- One valid review contract produces one `RENDER_RUNTIME` queue job.
- Duplicate submission does not create a second job.
- Queue payload contains the complete immutable handoff contract.
- Invalid checksum or malformed contract is rejected before queue mutation.
- Submission runtime never reads the active timeline store.
- Regression suite passes.
