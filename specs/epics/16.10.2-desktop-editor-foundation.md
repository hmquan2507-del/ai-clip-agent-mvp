# Sprint 16.10.2 — Desktop Editor Foundation

Date: 2026-07-24

## Context and course correction

A prior pass at this sprint (`specs/epics/16.10.2-editor-shell.md`, superseded by this document) took the approach of redesigning the *existing* Review editor's internal layout in place. This sprint's brief explicitly reverses that direction: Review UI is legacy and must be left alone; the Desktop Editor is a **brand new**, standalone UI surface that talks to the same runtime through an adapter layer.

Accordingly, the first step of this sprint was to **undo** the prior redesign:

- `frontend/src/features/review/shell/review-editor-shell.tsx` was restored to its original, pre-redesign content (`git checkout HEAD --`).
- `frontend/scripts/test-review-editor-shell.cjs` (a regression script that renders `ReviewEditorShell` and asserts on its markup) was restored to its original assertions.
- `frontend/src/features/editor-shell/` (the component tree built for the reverted approach) was deleted — it was disposable scaffolding from this session, not pre-existing product code.

Review UI now renders exactly as it did before either of the last two sprints touched it, and `/review` continues to work unchanged.

## Architecture

```
/editor/:productionId (page.tsx)
  └─ DesktopEditorRuntimeAdapter   (new — features/desktop-editor/runtime-adapter.tsx)
       └─ ReviewWorkspaceProvider   (existing, untouched — establishes the session)
            └─ DesktopEditorRuntimeContent   (new — mirrors runtime-workspace.tsx's wiring)
                 └─ DesktopEditorShell        (new — the entire new UI)
                      ├─ DesktopEditorHeader          (new, presentational)
                      ├─ EditorToolRail               (new, presentational)
                      ├─ EditorPreviewCanvas           (new wrapper) → ReviewPreviewStage (existing, unchanged, reused)
                      ├─ EditorInspector
                      │    ├─ "Properties" tab         (new, placeholder sections)
                      │    └─ "AI Copilot" tab → EditorAiCopilot (new cards) + ReviewAICommandBar (existing, unchanged, reused)
                      ├─ EditorAssetPanel              (new, presentational, mock data)
                      ├─ EditorTimelineWorkspace (new wrapper) → ReviewTimelinePanel (existing, unchanged, reused — real drag/trim/select wiring)
                      └─ EditorStatusBar               (new, presentational, read-only)
```

The Review UI's own composition root (`ReviewWorkspace` → `ReviewRuntimeWorkspace` → `ReviewEditorShell`) is completely separate and untouched — it is not imported by, and does not import, anything under `features/desktop-editor/`. The two UIs are independent consumers of the same underlying runtime.

`DesktopEditorRuntimeAdapter`/`DesktopEditorRuntimeContent`/`DesktopEditorRuntimeEditor` (all in `runtime-adapter.tsx`) are a **new** file that consumes the runtime's public API exactly the way `features/review/integration/runtime-workspace.tsx` does — `ReviewWorkspaceProvider`, `useReviewWorkspaceState`, `useReviewWorkspaceActions`, `buildReviewEditorViewModel`, `useReviewRuntimeClipDrag`, `useReviewRuntimeClipTrim`, `useReviewRuntimeKeyboardEditing` (all public exports of `@/features/review`/`integration/index.ts`, designed for reuse). No runtime file was opened for editing; the adapter only calls these hooks and renders `DesktopEditorShell` instead of `ReviewEditorShell` with the resulting data.

- `frontend/src/app/editor/[productionId]/page.tsx` — now renders `<DesktopEditorRuntimeAdapter productionId={...} />` instead of `<ReviewWorkspace productionId={...} />`. This is the only routing change; `/review` is untouched and still renders the legacy `ReviewWorkspace`.

- `frontend/scripts/test-workspace-foundation.cjs` — the `editor_reuses_review_runtime` check (written in an earlier sprint) asserted that the editor page's source contained the literal string `"ReviewWorkspace"`. Since the editor route intentionally no longer renders `ReviewWorkspace` directly (per this sprint's explicit "do not extend/reuse the Review layout" rule), the check was updated to assert the page renders `DesktopEditorRuntimeAdapter` instead — the same underlying invariant (editor route wired to the runtime), reflecting the now-correct architecture.

## Runtime integration

- **Preview**: `EditorPreviewCanvas` renders the existing `ReviewPreviewStage` unmodified, passed the same `view?.preview` view-model slice `ReviewEditorShell` uses.
- **Timeline**: `EditorTimelineWorkspace` renders the existing `ReviewTimelinePanel` unmodified, with the identical drag/trim/select/clipboard/command props the runtime adapter assembles from `useReviewRuntimeClipDrag`/`useReviewRuntimeClipTrim`/`useReviewRuntimeKeyboardEditing` — the same three hooks, called the same way, as in the untouched `runtime-workspace.tsx`.
- **AI**: The AI Copilot's custom-prompt box renders the existing `ReviewAICommandBar` unmodified, wired to the same `onAICommandSubmit`/pending/disabled/accepted-submission props as before. The ten preset suggestion cards (Highlight first 15s, Remove silence, Add B-roll, Replace music, Improve subtitles, Rewrite hook, Generate CTA, Translate subtitles, plus the custom prompt) are callback-only per spec — no API calls, no timeline mutation.
- **Header controls** (undo/redo/export/refresh) are wired to the same `actions.undoTimeline`/`redoTimeline`/export-render-contract flow the Review runtime adapter already used (copied verbatim from `runtime-workspace.tsx`'s export-workspace integration, since that logic lives in the adapter, not in a runtime file).
- **Error handling**: `DesktopEditorRuntimeContent` always renders `DesktopEditorShell` — even in the `idle`/`opening`/`closed`/`error` states — passing a `runtimeError` string only when there's no snapshot to show. `DesktopEditorShell` forwards that string exclusively to `EditorPreviewCanvas`, which is the only component that swaps to an error view; header, tool rail, asset panel, inspector, timeline, and status bar keep rendering normally. This was verified via a manual dev-server request to `/editor/:id`, whose initial (pre-hydration) HTML already contains `data-desktop-editor-shell="true"` — the full new shell chrome, not a full-page spinner.

## Known limitations

- Preview transport (play/pause/frame-step/zoom/aspect-ratio/fullscreen) is the same decorative, non-synced UI already present inside the reused `ReviewPreviewStage` — real playback sync remains a future dependency, unchanged by this sprint.
- `EditorAssetPanel` collection tabs (AI Assets, Stock, Photos, Music, SFX, Templates, Brand Kit) all show the same mock asset list; there's no per-collection data source yet.
- `EditorInspector`'s "Properties" tab is a pure placeholder (Video/Audio/Text/Animation/Transform/Effects/Transitions sections with no fields), per spec.
- The AI Copilot's nine preset suggestion cards are inert (callback prop with no default consumer) — intentionally, per the "callbacks only, no API, no timeline mutation" instruction.
- `EditorStatusBar`'s Zoom/Snap/Resolution/Speed fields have no backing live value in the current `ReviewTimelineView` contract and show fixed placeholder text; FPS, Duration, and Cursor are real.
- No resizable panels — no such library exists in `package.json` yet, and the brief only asked that panels be "prepared for resizing," not implemented.
- Desktop-only (`>=1400px` assumed); no mobile/tablet layout was built, per spec.
