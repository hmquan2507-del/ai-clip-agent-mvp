# Sprint 16.10.4 — AI Timeline Runtime

Date: 2026-07-24

A new `features/ai-timeline/` layer that sits **visually above** the existing, untouched Timeline Runtime and makes every AI editing decision visible, inspectable, and independently regeneratable. Visualization only — no runtime, backend, or business-logic changes.

## Architecture

```
Desktop Editor
  └─ Timeline Workspace (features/desktop-editor/components/editor-timeline-workspace.tsx)
       ├─ AiTimeline                (new — features/ai-timeline)          ← sits ABOVE
       └─ ReviewTimelinePanel       (existing, unchanged — Timeline Runtime)
```

`EditorTimelineWorkspace` (owned by this repo's UI layer, not the runtime) now renders `<AiTimeline duration={view?.duration ?? 0} />` in a fixed-height (140px), independently-scrolling region directly above the unchanged `<ReviewTimelinePanel {...panelProps} />`. `AiTimeline` receives only a `duration` number (read from the real timeline's own view model, `ReviewTimelineView.duration`) and an optional `onBlockAction` callback — it has no import of, and makes no call into, anything under `features/review/state`, `features/review/integration`, `features/playback`, the backend, or the AI runtime. The "Runtime Adapter" hop described in the brief's architecture diagram is exactly this: `AiTimeline` reads one number the runtime already exposes and renders independently of it.

`AiTimeline` also never owns playback or editing logic: clicking a block only updates the AI Timeline's own local `selectedBlockId`/`hoveredBlockId` state and (optionally) calls `onBlockAction`; nothing here calls `useReviewWorkspaceActions`, moves a clip, or touches the timeline's session/history state.

## AI Block model

```ts
interface AiBlock {
  id: string;
  trackKind: AiTrackKind;      // one of the 10 track kinds below
  title: string;
  startTime: number;
  endTime: number;
  confidence: number;          // 0..1
  reason: string;
  source?: string;
  aiModel: string;
  generatedAt: string;
  promptUsed?: string;
  estimatedImpact?: string;
  linkedClipId?: string;       // future cross-highlight hook, unused this sprint
  pinned?: boolean;
  disabled?: boolean;
  manual?: boolean;
  // + track-kind-specific extras: cutReason, emotionLevel, storyBeat,
  //   wordEmphasis, animation, zoomIntensity, durationRemoved, hookStrength
}
```

Every block renders: icon + colored border/fill (by track kind), title, time range (via position/width), confidence (in the tooltip), reason (tooltip + Explain dialog), source/AI model/prompt/estimated impact (tooltip). Near-instant decisions (duration < 0.4s — emoji, cut points) render as a diamond `AiMarkerView` instead of a ranged `AiBlockView` bar.

| Track (in order) | Filterable | Notes |
|---|---|---|
| Hook Detection | Hook | hook strength, confidence |
| AI Cuts | — | reason: silence / emotion / scene-change / hook / cta / visual-interest |
| B-roll | B-roll | reason, source, duration, confidence |
| Subtitle Highlight | Subtitle | word emphasis, animation |
| Zoom | Zoom | intensity, duration, reason |
| Emoji | — | point markers |
| CTA | CTA | placement |
| Silence Removed | Silence | duration removed, reason |
| Emotion Peak | Emotion | high / medium / low |

Every track: independently **shown/hidden**, **locked/unlocked**, **muted/unmuted** (`AiTrackHeader`'s three icon toggles), and **collapsible** (chevron → collapses to a 24px header-only row). State lives in `useAiTimelineState`'s `trackStates` map — pure UI state, not persisted to any runtime.

## Interactions

- **Single click** a block → `state.setSelectedBlockId`, giving it a highlighted outline. (Cross-highlighting the corresponding real timeline clip is a future dependency — see below.)
- **Double click** → opens the Explain dialog (`AiRegenerateDialog` in `explain` mode).
- **Hover** → `AiTooltip` follows the pointer, showing AI Model / Generated At / Reason / Confidence / Prompt Used / Estimated Impact; a vertical `AiConnectionView` guide line is drawn through every visible track at that block's start time, and the hovered block's own track lane brightens.
- **Right-click** → a context menu (`Regenerate`, `Disable`, `Duplicate`, `Convert to Manual`, `Delete`, `Pin`, `Explain`). `Pin`/`Disable`/`Convert to Manual` toggle a local `blockOverrides` map instantly (cosmetic-only state owned by the AI Timeline itself — not a real edit). `Regenerate`/`Explain` open the shared dialog. `Duplicate`/`Delete` call `onBlockAction` only — no local mutation, since those conceptually create/remove a real AI decision, which is out of scope for a visualization-only sprint.
- **Explain** dialog quotes the block's own `reason` string (e.g. *"1.4s of silence reduced pacing"*).
- **Regenerate** dialog explicitly scopes to the one block passed in — the confirm button's label includes the block's own title, and the body text states the rest of the production is untouched. There is no "regenerate everything" action anywhere in this feature.

## Search, filters, mini map

- **Search** (`state.search`) matches block `title`/`reason` (case-insensitive); matches are highlighted (non-matches dim to 35% opacity) and `Enter` scrolls the shared viewport to the first match (`viewport.scrollToTime`).
- **Filters** (`Hook`/`Subtitle`/`B-roll`/`CTA`/`Emotion`/`Zoom`/`Silence` chips) — when any filter is active, only tracks whose `filterKey` matches an active filter render; clearing all filters restores every track to its own shown/hidden state.
- **Mini map** (top-right of the toolbar) — a compact 160×20px strip spanning the full production duration, plotting one dot per visible block (colored by track) plus a draggable-by-click viewport window reflecting the AI Timeline's current horizontal scroll/zoom. Clicking anywhere on it seeks the viewport there.

## Performance

- **Windowing**: `AiTrackRow` only renders blocks whose time range intersects the current visible range (± a 5s buffer) — computed once per scroll/zoom change via `useMemo`. This is the mechanism that keeps "thousands of AI blocks" smooth: off-screen blocks are never mounted, not merely visually clipped.
- **Memoization**: `AiTrackRow`, `AiBlockView`, `AiMarkerView`, and `AiConnectionView` are all wrapped in `React.memo`; `useAiTimelineState`, `useAiTimelineViewport`, and `useAiTimelineMockData` all return `useMemo`d objects so consumers don't re-render on unrelated parent updates.
- **Container sizing**: a `ResizeObserver` on the scroll container drives `viewport.containerWidth` (used for windowing math and the mini map's viewport box) — the natural tool for "size changed externally" rather than a resize-drag gesture (which the prior sprint's `layout-resizer.tsx` already covers for panel dimensions).
- No external virtualization library was added — the windowing is implemented directly (filter-by-visible-range), keeping the dependency footprint unchanged, consistent with this codebase's preference to avoid new heavy dependencies unless clearly necessary.

## Known limitations

- **Cross-panel synchronization is partial.** Hovering a block highlights that block and draws a guide line across the AI Timeline's own tracks, but it does not (yet) highlight the corresponding clip in `ReviewTimelinePanel`, the Preview canvas, or the Inspector — those live in separate feature trees (`features/review`, `features/desktop-editor`), and wiring true cross-highlighting would mean either lifting shared "current AI focus" state up into `DesktopEditorShell` (straightforward, not done this sprint) or reading into the real timeline's selection state (would require touching Selection Runtime, explicitly out of scope). `AiTimeline` accepts an `onBlockAction`/hover-adjacent extension point ready for that future wiring.
- **The AI Timeline's horizontal scroll/zoom is independent of the real timeline's.** `ReviewTimelinePanel` owns its own internal viewport (via the untouched `useReviewTimelineViewport`), which isn't exposed in a form the AI layer could safely share without duplicating runtime state. `AiTimeline` therefore keeps its own `useAiTimelineViewport` — visually "above" the real timeline as required, but not pixel-synced to its zoom/scroll position. Both timelines represent the same total `duration`, so they stay proportionally aligned, but a zoom action in one does not move the other.
- **Data is entirely mock.** `useAiTimelineMockData` generates a fixed set of ~23 seeded decisions proportional to whatever `duration` is passed in; there is no AI runtime or backend call. This is intentional per the sprint's "visualization only" scope.
- **Duplicate/Delete are inert** beyond calling `onBlockAction` — no handler is wired yet (no consumer of `AiTimeline`'s `onBlockAction` prop exists outside this feature), so they currently just close the context menu. Pin/Disable/Convert-to-Manual do have an immediate local visual effect since those are reasonably "cosmetic" (not a real edit).
- **Track locking (`locked`) is display-only** — there's no actual interaction on a track's blocks to lock against yet (blocks are always clickable/draggable-context-menu-able regardless of the lock icon's state); it's wired to the icon UI so a future sprint can gate accordingly.
