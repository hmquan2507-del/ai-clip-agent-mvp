# Sprint 16.10.5 — AI Timeline Synchronization Runtime

Date: 2026-07-24

Transforms last sprint's AI Timeline **visualization** into a **synchronized layer** sharing one coordinate system with the real (untouched) Timeline Runtime — without rebuilding, forking, or editing it.

## The core constraint, and how it was resolved

Before writing any code, I traced exactly what the real Timeline Runtime (`features/review/`) does and does not expose, since "synchronize with it, don't touch it" only has one honest reading once you know the actual boundary:

| Value | How it's obtained | Mechanism |
|---|---|---|
| `duration`, `playheadTime`, `revision`, selected clip ids | **Plain props** — already part of `ReviewEditorViewModel`/`ReviewTimelineView`, built once by the untouched, publicly-exported `buildReviewEditorViewModel(state)` and threaded down to both `ReviewTimelinePanel` and this app's own components | Zero DOM tricks needed — genuinely the same live values, same source of truth |
| Zoom level, scroll position, real content width | `useReviewTimelineViewport` is called **inside** `ReviewTimelinePanel` itself, with fully local `useState`-per-call-site state — there is no prop, callback, context, or store to subscribe to | **Read-only DOM observation**: the scroll container carries a live `data-review-timeline-zoom` attribute (a `MutationObserver` target) and is itself a real scrollable DOM node (`scrollLeft`/`scrollWidth`/`clientWidth` — native browser properties, no library needed) |

This split is the whole architecture. `TimelineViewportContext` (below) exposes both kinds of value through one interface, but is honest internally about which are "real props, zero risk of drift" and which are "observed, read-only, and depend on the real timeline's current (undocumented) DOM contract."

## Architecture

```
Desktop Editor
  └─ Timeline Workspace (EditorTimelineWorkspace)
       └─ TimelineViewportProvider   (new — features/ai-timeline/context)
            │   props: duration, playheadTime, revision, selectedClipIds
            │   (all sourced from the SAME ReviewEditorViewModel already
            │    flowing into this component — no new data path)
            │
            ├─ AiTimeline                        (rewritten — no more independent viewport)
            │     reads: pixelsPerSecond, scrollLeft, timelineWidth,
            │            visibleRange, zoomLevel, playheadTime, duration,
            │            revision, selection — ALL from context, none owned locally
            │
            └─ TimelineViewportObservedRegion      (new — thin ref wrapper)
                  └─ ReviewTimelinePanel            (UNCHANGED, Timeline Runtime)
                        └─ its scroll container is what the Provider observes
```

`TimelineViewportProvider` never imports, calls, or re-implements anything from `features/review/state`, `features/review/integration`, `features/review/viewport`, or `features/playback`. It only (a) receives already-public prop data, and (b) attaches a `MutationObserver`/native `scroll` listener/`ResizeObserver` to a DOM node it locates via `querySelector('[data-review-timeline-scroll]')` inside a wrapper `<div>` it's given a ref to — the same category of interaction a browser extension or accessibility tool could do, not a code-level dependency.

## Viewport flow

```
ReviewTimelinePanel's real scroll container (DOM)
  │  native `scroll` event, `data-review-timeline-zoom` attribute mutation, `scrollWidth`/`clientWidth`
  ▼
TimelineViewportProvider (rAF-throttled read: scrollLeft, contentWidth → pixelsPerSecond, zoomLevel, timelineWidth)
  │  React context value (memoized)
  ▼
AiTimeline: pixelsPerSecond × block.startTime/endTime → block position (exactly like real clips)
AiTimeline's own scroll container: scrollLeft kept mirrored to context.scrollLeft (controlled effect)
AiMiniMap: viewport window box computed from the same visibleRange
```

**Bidirectional scroll**: when a user scrolls the AI Timeline, its `onScroll` calls `viewport.setScrollLeft(next)`, which writes `realNode.scrollLeft = next` directly (a native DOM property write — not a call into any runtime function) and updates local context state. The real timeline's own `onScroll` handler (already wired internally, untouched) fires in response and updates its own internal state exactly as if the user had scrolled it directly. Setting the same value twice is a no-op, so this round-trip is stable, not a feedback loop.

## Synchronization flow (playhead / selection / hover / stale)

- **Playhead**: `viewport.playheadTime` (a real, live value) drives a persistent vertical guide line (`AiConnectionView`, red) across every AI track and a matching tick on the mini map — updates every time the underlying view model updates, no separate polling.
- **Selection (real → AI)**: `viewport.selection` (the real `clipboard.selectedClipIds`, live) is compared against each `AiBlock.linkedClipId` (assigned at mock-data-generation time by matching a seeded block's time range against the real clips present when the production loaded) — a selected real clip visually selects its matching AI block(s), with zero AI-owned selection logic.
- **Selection (AI → real)**: clicking an AI block calls `onRequestSelectClip(linkedClipId)`, a plain callback bubbled up to `EditorTimelineWorkspace`, which invokes the real, untouched `onSelectClip` callback already threaded through from the runtime adapter (`{ clipId, additive: false, moveCursor: true }`) — the AI layer never calls a selection action itself, it only asks the existing pathway to do it.
- **Hover**: hovering an AI block draws the existing guide line at that time position (now using the shared `pixelsPerSecond`, so it lines up with the real timeline underneath) and shows the tooltip; there is no reverse direction (real clip hover → AI highlight) because `ReviewTimelinePanel` exposes no hover callback or stable per-clip DOM selector — documented below rather than faked.
- **Multi-selection**: since `viewport.selection` is a `Set`, selecting several real clips highlights every AI block whose `linkedClipId` is in that set — no special-casing needed, it's the same single-block logic applied per block.
- **Stale detection**: each `AiBlock` is stamped with `generatedAtRevision` (the real timeline's revision at the moment mock data was generated). Live `viewport.revision` is compared against it every render; a mismatch marks the block `stale` (dashed border) without removing or silently regenerating it. A revision banner appears with **Refresh** (bulk-stamps affected blocks to the current revision via the existing per-block override mechanism), **Keep** (dismisses the banner for this revision only — reappears if revision changes again), and **Compare** (opens the Explain dialog on the first stale block — a deliberately modest stand-in; see limitations).

## AI decision states

`resolveAiBlockVisualState()` (in `types.ts`) derives one mutually-exclusive visual state per block per render, in priority order: `processing` → `regenerating` → `disabled` → `stale` → `selected` → `hovered` → `pinned` → `normal`. Each renders distinctly (pulse animation for processing/regenerating, dashed border for stale, solid ring for selected, etc.) in both `AiBlockView` and `AiMarkerView`.

## Revision indicator

The AI Timeline's toolbar now shows a compact `Rev {N}` chip with a connection dot (green when the DOM observation has successfully attached to the real timeline's scroll container, amber otherwise) and a stale-count badge when applicable — plus the dismissible/refreshable banner described above.

## Search (jump → center → flash)

`Enter` in the search box now (a) scrolls the shared viewport so the first match is **centered** (`scrollLeft = matchStart × pixelsPerSecond − timelineWidth / 2`, clamped ≥ 0) instead of just visible, and (b) sets a 900ms `flashBlockId` that's folded into the same highlight mechanism used for search-matching, giving the matched block a brief flash before returning to normal.

## Mini map

Now also shares the real viewport: its viewport-window box and dot positions are computed from `viewport.visibleRange`/`viewport.duration`/`viewport.pixelsPerSecond` (previously an independent local scroll/zoom), and it now also renders the shared playhead tick. Clicking it calls the same `setScrollLeft` used everywhere else — one relay path, no separate scrollbar/scroll state.

## Performance notes

- **No duplicated arrays**: `AiTimeline` still computes `blocksByTrack` via `useMemo`, but position math for every block reads the single shared `pixelsPerSecond`/`scrollLeft` from context — there's exactly one viewport number set, not one per track or one per component.
- **No duplicated playhead**: the playhead value is read once from context and passed straight into a single `AiConnectionView`; nothing re-derives or re-fetches it.
- **rAF-throttled DOM reads**: the provider's `MutationObserver`/`scroll` callback schedules at most one `requestAnimationFrame` read per animation frame, coalescing bursts of scroll events instead of reading `scrollWidth`/`clientWidth` synchronously on every event.
- **Memoized selectors**: `visibleRange`, `selection` (as a `Set`), and the provider's context value are all `useMemo`d; `AiTrackRow`/`AiBlockView`/`AiMarkerView`/`AiConnectionView` remain `React.memo`-wrapped from the prior sprint.
- **Windowing unchanged and still correct**: block windowing in `AiTrackRow` already filtered by `visibleRange`, which now reflects the *real* timeline's zoom/scroll instead of an independent one — so windowing precision only improved.
- Target "60 FPS scrolling" is supported by the above (rAF batching + memoization), though it wasn't independently profiled in this sandbox (no browser available here — see Known Limitations in the prior sprint's report for the same caveat).

## Known limitations

- **Hover/select on the real timeline does not highlight AI blocks (one direction only).** `ReviewTimelinePanel` exposes no hover callback and no stable per-clip DOM selector (`data-clip-id` or similar) — only `aria-label`s containing a Vietnamese label string, not a stable id. Adding one would be a minimal, low-risk change to `shell/timeline.tsx`, but it is still an edit to a file this sprint (and the two before it) was explicitly told not to touch, so it's intentionally left undone rather than worked around unsafely.
- **The zoom/scroll bridge depends on an undocumented DOM contract.** `data-review-timeline-zoom`/`data-review-timeline-scroll` and the scroll container's `scrollWidth`/`clientWidth` are current implementation details, not a published API — if a future change to `ReviewTimelinePanel` renames or removes them, `TimelineViewportProvider`'s zoom/scroll observation (not its playhead/revision/selection, which come from stable props) would silently stop connecting (`connected` would read `false`, and `AiTimeline` would fall back to `pixelsPerSecond = 0` gracefully rather than crash — but genuine sync would be lost until the selector is updated).
- **`linkedClipId` assignment is best-effort and one-shot.** It's computed once when mock blocks are generated (matching a seeded block's time range against whatever real clips existed then); if clips are later added, removed, or retimed, existing links are not re-resolved (this is arguably correct — it's exactly the kind of drift "stale" should surface — but there's no explicit UI yet distinguishing "stale because of a link mismatch" from "stale because of a revision bump").
- **"Compare" (in the stale banner) is a simplified stand-in** that opens the Explain dialog for the first stale block rather than a dedicated before/after diff view — no such component was in this sprint's scope, and building one wasn't warranted by the acceptance criteria's actual wording ("Compare" is listed as a user choice, not a specified UI).
- **60 FPS scrolling was not independently profiled** — this sandbox has no browser to measure real frame timing; the implementation follows the rAF-throttling/memoization patterns that make it achievable, but the number itself is unverified here.

## Future runtime dependencies

- If/when the Timeline Runtime team is willing to add a single `data-clip-id` attribute to each clip's DOM node (a presentational, additive, low-risk change), true bidirectional clip↔block DOM highlighting becomes straightforward using the same `TimelineViewportObservedRegion` pattern already built here.
- A published, versioned viewport-state export from `features/review/viewport` (even read-only) would let this sprint's DOM-observation bridge be replaced with a direct subscription — removing the "undocumented DOM contract" risk noted above.
- Extend `linkedClipId` resolution to re-run (not just at generation time) once there's a real AI backend generating decisions against a live, changing clip list, so links can track edits instead of only detecting drift as staleness.
